# Основные пользовательские сценарии для проверки

## Контекст

Система поддерживает 2 роли без авторизации:
- Владелец календаря (`admin` часть)
- Гость (`public` часть)

Все сценарии ниже проверяются как интеграционные e2e-тесты через UI.

## Сценарии гостя

### GS-01: Просмотр типов событий владельца

**Предусловия:**
- Backend запущен
- Владелец `name-owner` существует

**Шаги:**
1. Открыть `/:ownerSlug` (`/name-owner`)
2. Дождаться загрузки списка типов событий

**Ожидаемый результат:**
- Отображается карточка владельца
- Отображается минимум 2 типа события (`Встреча 15 минут`, `Встреча 30 минут`)

### GS-02: Выбор типа события и просмотр слотов

**Шаги:**
1. На странице `/name-owner` нажать карточку `Встреча 15 минут`
2. Проверить переход на `/:ownerSlug/:eventTypeId`

**Ожидаемый результат:**
- Открывается экран бронирования
- Появляются календарь и список слотов
- В списке слотов есть статусы `Свободно`/`Занято`

### GS-03: Успешное бронирование свободного слота

**Шаги:**
1. Выбрать доступный слот
2. Нажать `Продолжить`

**Ожидаемый результат:**
- Показывается сообщение `Бронирование создано`
- Слот перестаёт быть доступным после обновления списка

### GS-04: Конфликт бронирования слота

**Шаги:**
1. Открыть одну и ту же страницу бронирования в двух вкладках
2. В первой вкладке забронировать слот
3. Во второй вкладке попытаться забронировать тот же слот

**Ожидаемый результат:**
- Во второй вкладке возвращается ошибка конфликта (`SLOT_CONFLICT`)
- В UI показывается сообщение о занятости слота

## Сценарии владельца (админка)

### AS-01: Просмотр списка типов событий

**Шаги:**
1. Открыть `/admin`

**Ожидаемый результат:**
- Виден раздел `Типы событий`
- Отображаются предзаданные типы событий

### AS-02: Создание нового типа события

**Шаги:**
1. На `/admin` заполнить форму (`id`, `name`, `description`, `durationMinutes`)
2. Нажать `Создать`

**Ожидаемый результат:**
- Новый тип события появляется в списке

### AS-03: Просмотр предстоящих встреч

**Шаги:**
1. После создания бронирования гостем открыть `/admin`
2. Проверить таблицу `Предстоящие встречи`

**Ожидаемый результат:**
- В таблице отображается бронь с корректным временем и гостем

## Нефункциональные проверки

- CORS preflight работает для frontend origin
- Сервис стартует без внешней БД (in-memory)
- После рестарта backend состояние очищается
